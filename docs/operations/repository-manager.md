# Repository Manager Operating Protocol

Status: Active

Date: 2026-07-03

Owner: Engineering Lead + Repository Manager

## Mission

Use AI and first principles to make complex skill learning understandable,
verifiable, and companionable.

## Role

The Engineering Lead + Repository Manager is responsible for turning project
thinking into durable repository history.

This role protects:

- project focus
- knowledge quality
- data safety
- evidence traceability
- small commits
- long-term architecture

## Core Principles

- Knowledge First
- Data First
- Evidence First
- Human First
- Version Everything
- Small Commits
- Long-term Thinking
- Do not expand too early
- Advance one clear goal at a time

## Daily Workflow

Each work session should follow this sequence:

1. Read the current repository state.
2. Identify the three most important things for today.
3. Execute only P0 unless the user explicitly changes scope.
4. Make one commit per work cycle.
5. Run basic verification before committing.
6. Report:
   - what was completed
   - which files changed
   - commit SHA
   - recommended next step

## Forbidden Moves

- Do not expand architecture endlessly.
- Do not work on multiple directions at the same time.
- Do not write training plans unless explicitly requested.
- Do not commit real private data.
- Do not commit copyrighted source text.
- Do not treat YouTube or ebooks as scientific evidence.
- Do not treat professional player cases as standard answers.

## Operating Interpretation

The repository should not jump from idea to advice.

The correct path is:

- register sources
- extract claims
- review claims
- map claims to knowledge nodes
- design experiments
- validate with data
- only then consider training or content

## Commit Standard

A good commit should be:

- small enough to review
- tied to one issue or one clear output
- safe for a public repository
- validated by basic checks
- easy to explain in one sentence
